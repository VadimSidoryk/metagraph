## Purpose

Read `scratch.md`, classify each unprocessed entry, and propose graph nodes to the user. After user confirmation, create the approved nodes as proper C/T-series entities. Clear processed entries from scratch.

This is the **classification layer** — it receives output from [[P1003]] (email/Slack filter) and manual notes added by the user throughout the day.

---

## Protocol

### Step 1 — Read scratch.md

Read `/metagpraph/scratch.md`. If the file is empty or missing, log a no-op run and exit.

Identify **unprocessed entries**: lines that do not have a `✅` or `🗑️` marker at the end.

### Step 2 — Classify each entry

For each unprocessed entry, determine the most likely node type:

| Signal in entry | Proposed type |
|---|---|
| Action for you to do, deadline, "нужно", "сделать" | Task → `T-series` in `tasks/todo/` |
| New fact, reference, external link, "узнал что", "заметка по теме" | Knowledge node → `C-series/knowledge/` |
| Decision made or needed | Decision log entry → `C-series/decision/` |
| Goal-related update, OKR progress | Update to existing goal node |
| Low signal, noise, no clear category | Discard → `🗑️` |

Also check: does this entry reference a person in `context/personal/`? If yes, link the proposed node to their personal node.

Does it reference an active meeting from the calendar context in the entry? If yes, note it as context in the task/node description.

### Step 3 — Present proposed triage to user

Format the proposal as a clear, scannable list. For each entry:

```
[Entry text]
→ Предлагаю: [T1001 — задача "Подтвердить бюджет с Антоном" | приоритет: high | дедлайн: 2026-04-04]
   Контекст: [[P1003]], встреча [Q2 Planning]
```

For entries proposed as discard:
```
[Entry text]
→ Предлагаю: удалить (нет явного действия или знания)
```

Ask the user to confirm or correct each item. The user can:
- Approve (`да` / `+`) — create as proposed
- Correct (`задача, но дедлайн пятница`) — adjust and create
- Skip (`нет` / `-`) — mark as discarded without creating a node

### Step 4 — Create approved nodes

For each approved item:
1. Read `INDEX.md` → get next available ID (T or C series)
2. Update `INDEX.md` immediately (increment counter, register entity)
3. Create the node file in the correct folder using the template from `CLAUDE.md`
4. Fill frontmatter: `creator: process:P1004`, `context_refs` referencing source meeting/person if available
5. Note in `## Graph Changes` which scratch entries were processed

### Step 5 — Mark entries in scratch.md

After processing:
- Approved and created → append `✅ → [[T1001]]` to the line
- Discarded → append `🗑️` to the line

This ensures re-runs don't re-process the same entries.

### Step 6 — Auto-archive stale entries

Any entry older than **14 days** that is still unprocessed → automatically mark as `🗑️ [auto-archived: >14d]` without asking the user.

### Step 7 — Capacity check

If scratch.md has more than **20 unprocessed entries** → prepend a warning at the top of the triage proposal:
```
⚠️ scratch.md содержит X необработанных записей (>20). Рекомендую разобрать или сбросить старые.
```

### Step 8 — Log the run

Append to **Execution History**:

| Date | Entries read | Created (T) | Created (C) | Discarded | Stale auto-archived | Notes |
|---|---|---|---|---|---|---|

### Step 9 — Commit

Commit all created node files + updated scratch.md + updated INDEX.md in a single commit:
```
sync(P1004): triage scratch — N tasks, M knowledge nodes created
```
